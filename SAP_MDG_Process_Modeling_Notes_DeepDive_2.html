What is it

Governance Scope — the boundary of MDG's authority

Governance Scope defines what MDG governs and how far its authority reaches. It is the answer to the question: "Which master data, and which operations on that master data, fall under MDG's control?"

In plain words, governance scope sets the boundary line: anything inside the scope must follow the rules of MDG (create through a CR, get approved, get activated). Anything outside the scope is left alone for normal SAP transactions to handle.

Y
Why we use it

Why scope must be defined

Not every piece of data in an SAP system is master data. Not every piece of master data needs heavy governance. By defining a clear scope, MDG:

  • Picks battles wisely — only the high-value master data (material, vendor, customer, finance) gets the full governance treatment.
  • Avoids friction — low-risk transactional data is not forced through approval workflows.
  • Targets the right operations — only the operations that change data (create, update, delete, block) need governance; pure reads do not.
  • Aligns with the organization's policies — governance scope can be tuned per company: one organization may govern only vendors, another may govern everything.
F
What governance covers

The universe of operations under MDG control

Governance scope typically includes the following operations on master data — these are exactly the operations that need a Change Request:

Operations inside governance scope
Create
Bring a new master data record into existence.Example: a new vendor "ACME Corp" is onboarded.
Change / Process
Modify fields on an existing record (description, address, bank details, tax data).Example: a vendor changes their bank account number.
Delete / Mark for Deletion
Flag a record so it can no longer be used in transactions.Example: a discontinued material is marked for deletion.
Block / Unblock
Temporarily prevent use of a record without deleting it.Example: pause purchases from a vendor under legal review.
Mass Processing
Apply the same operation across many records at once.Example: bulk update unit of measure for 200 materials.
Copy
Create a new record using an existing one as the starting template.Example: duplicate a vendor record for a new region.
Cleanse
Fix data quality issues (standardize formats, remove duplicates).Example: standardize address format across all customers.

If governance scope did not exist, MDG would either be everywhere (slowing every tiny transaction) or nowhere (no rules at all). A clearly defined scope is what makes MDG practical — it lets the organization decide "these data domains and these operations are important enough to govern, and everything else continues normally."

Think of traffic rules in a city. The traffic department doesn't govern every footpath — it focuses on roads, highways, and specific zones. Inside those zones, you need a license, a registered vehicle, and you must follow signals. Outside the zones (your own backyard), you can ride a bicycle however you want.

Governance scope is the same idea: MDG decides which "roads" of master data need rules (license = CR, signals = workflow, registration = approval), and which areas need no special treatment.

L
How governance scope is realized

The IMG path where scope is configured

Governance scope is not configured as one single field. It is the combined effect of several settings:

  • Data models you enable — picking MM, BP, 0G, 0F decides which master data domains are in scope.
  • Change Request Types you define — each CR Type adds a specific operation (create material, change vendor, etc.) to the governed scope.
  • Business activities you create — each business activity adds an allowed verb (create, change, mass) to the scope.
SPRO→Master Data Governance→General Settings→Process Modeling

Everything that follows in this guide — Change Request, Header, Logical Action, Business Activity, CR Type, Steps, Workflow — is the machinery that implements governance scope.

What is it

The Change Request (CR)

A Change Request is a formal request a user creates whenever they want to do anything to master data in MDG — create a new material, change a customer's address, delete a vendor, block a business partner. It is the only way master data can be touched in the MDG system.

Think of it as a "sealed envelope with the proposed changes inside." The system holds your changes inside this envelope, routes it for approval, and only writes them to the database after every approver has signed off.

Y
Why we use it

Why does MDG force every change through a CR?

  • Governance & control — no one can sneak a bad value into production master data. Every change is reviewed by at least one approver.
  • Audit trail — who created what, who approved, when, and why. The CR carries this history forever, which is essential for compliance audits (SOX, GDPR).
  • Workflow routing — the CR is the package that gets shipped from person to person until approved. Without the CR, there would be nothing to route.
  • Validation & rollback — bad data can be caught mid-flight and rejected without ever polluting the active tables.
  • Different operations on the same data — create, update, delete, mass-update can all live as separate CR types for the same data model. The CR carries the "intent" along with the data.
F
Anatomy of a CR

What a Change Request actually contains

A CR is more than just a form. It is a structured object with three layers:

The three layers of a Change Request
1. Header
Metadata about the CR — CR number, CR type, requester, current step, status, workflow template. (Section 03 covers this in detail.)
2. Data Payload
The actual master data values being proposed — material number, vendor name, customer address, etc. This is what eventually gets written to active tables if approved.
3. History & Comments
Every action taken on this CR — who submitted, who approved, who rejected, what comments they added. Permanent audit log.
F
Lifecycle states

The states a CR moves through

CR status values you will encounter
Draft
The requester is still filling in details. Not yet submitted.
In Processing
Submitted, currently sitting at one of the workflow steps awaiting action.
Approved
All approvers have signed off. Ready for final activation.
Activated
The data has been written to the active tables. The CR is essentially "live."
Rejected
An approver said no. CR is closed without applying any changes.
Withdrawn
The requester pulled it back. CR is closed.
In Revision
Sent back to the requester to fix something (step 91/92/93/95).

If change requests didn't exist, anyone with a transaction code (like MM01) could create or change master data directly. There would be no review, no approvers, no record of who changed what or why. Quality of master data — the lifeblood of every SAP report — would collapse within months. MDG's entire reason for existing is to put a CR between users and the master data tables.

Think of buying a property. You don't just walk into the land office and write your name on the deed. You file an application (the CR), it goes to the registrar (workflow), then the legal officer (approver), then the sub-collector (final approval), and only then your name is entered in the official record (active table).

Each handoff is a workflow step. The application is the change request. The final stamp is the activation.

L
What it links to

The CR's neighborhood

  • Up: the user / requester who created it.
  • Sideways: the Change Request Type that defines its rules (data model, what it can do, which workflow drives it).
  • Through: the Workflow Template (usually WS60800086) that moves it from approver to approver.
  • Down: the master data record(s) it will eventually create or modify.

What is it

The CR Header

The Change Request Header is the top-level information that describes the CR — like the cover page of a file. It tells the system what kind of request this is, which data model it touches, what workflow will drive it, and what UI screen the user should see.

You do not configure the header directly — you configure it indirectly by configuring the Change Request Type. The header is the runtime instance; the CR Type is its design-time blueprint.

Y
Why it matters

The header is the system's instruction sheet

When MDG receives a new CR, it reads the header first to decide:

  • "Which data model do I touch?" (e.g. MM for material)
  • "Is this a create, change, delete, or mass operation?" (via business activity → logical action)
  • "Which workflow do I launch?" (workflow template number)
  • "Which screen do I show the user?" (UI application + UI configuration)
  • "Is this single-record only or multi-record?" (single-object flag)

Without these answers the CR cannot move an inch.

If the header didn't carry these fields, every CR would be a blob of unknown data — the system wouldn't know whether you want to create a customer or delete a material, which screen to show you, or whom to send it to. The header is the system's only way to interpret what the CR means.

What is it

The Logical Action

A Logical Action is SAP's standardized name for the type of operation you want to perform on master data. It is the verb of every CR: CREATE, CHANGE, DELETE, BLOCK, DISPLAY, MASS, COPY, CLEANSE, LOAD.

SAP ships these — you do not invent your own. They are global to all data models. A CREATE means the same thing whether you are creating a material, a customer, or a cost center.

Y
Why we use it

It is the universal action language

  • Standardization — the system handles every CREATE the same way internally, regardless of data model.
  • Routing logic — workflows can ask "is this a CREATE or a DELETE?" and route differently based on the answer.
  • Validation — different rules apply to creates vs deletes (e.g., creates need uniqueness checks; deletes need dependency checks).
  • UI behavior — fields may be editable for CREATE but read-only for DISPLAY.
U
How it is used

Where it sits in the chain

You never attach a logical action directly to a CR Type. The path is:

Logical Action→attached to→Business Activity→attached to→Change Request Type

So the logical action's job is to give the Business Activity its meaning. The Business Activity is just a label — the logical action is what tells the system "this label means CREATE."

F
Standard logical actions

The 9 verbs you'll see

SAP-shipped logical actions
CREATE
Generate a brand-new master data record. The system checks the key does not already exist, then inserts.Used by MAT1 for material creation.
CHANGE
Modify an existing record. The system fetches the current record and applies your edits.Used by MAT2 for material change.
DISPLAY
Read-only view of a record. No save allowed.Used by MAT3 for material display.
DELETE
Hard delete a record (rare for master data — usually a mark-for-deletion instead).Used by MAT6.
BLOCK
Logically block a record so no new transactions can use it, but the data stays in the system.e.g. blocking a vendor under fraud review.
MASS
Apply the same operation to many records in one CR.Used by MATA — process multiple materials.
COPY
Duplicate an existing record as a starting point for a new one.e.g. clone material M1 → M2 with minor changes.
CLEANSE
Data-quality cleanup — standardize, normalize, deduplicate.Used in MDG-C, MDG-S cleansing scenarios.
LOAD
Bulk load master data from outside (e.g. file upload, migration).Used in initial load programs.

Without logical actions, the system would have no idea whether your CR is supposed to create something new, modify something old, or delete something. Every CR would look identical to the engine. There would be no way to apply different validations, different UIs, or different workflows for different operations.

Imagine a hospital with a single form for everything — admission, discharge, surgery, observation. Doctors and nurses wouldn't know what to do with the patient. By stamping ADMIT, DISCHARGE, OPERATE on the form, the hospital instantly routes the patient through the right process. Logical actions are those stamps.

What is it

The Business Activity

A Business Activity is a configurable record that binds a Logical Action (the verb — e.g. CREATE) to a Business Object Type (the noun — e.g. Material/194). It is the layer where MDG learns that "this combination is allowed and meaningful."

The naming convention is consistent: MAT1 = Material + Create, MAT2 = Material + Change, MAT3 = Material + Display, MAT6 = Material + Mark-for-Deletion, MATA = Material + Mass.

Y
Why we use it

Why not just use Logical Action directly?

Because logical actions are global verbs. They don't say what you are creating. CREATE is the same word whether you're creating a material, a vendor, a cost center, or a bank account. The Business Activity adds the object context:

  • MAT1 = CREATE + Material → so the CR knows to use material fields, material validations, material UI.
  • BP1A = CREATE + Business Partner → so the CR uses BP fields, BP validations, BP UI.
  • The same logical action (CREATE) is reused across both — only the object changes.

This is the layer of reuse and combination. Without it, SAP would have to ship CREATE_MATERIAL, CREATE_VENDOR, CREATE_BP, CREATE_COST_CENTER as separate hard-coded verbs. Instead, it has 9 verbs × many object types → unlimited business activities.

L
What it links to

Two upstream, one downstream

Three ingredients combine here:

  1. Business Object Type — comes from the data model (e.g. 194 for material). This is yesterday's data-modeling output.
  2. Logical Action — chosen from the SAP-delivered list (CREATE / CHANGE / etc.).
  3. Description — a friendly label.

And one downstream:

  • Change Request Type — your Business Activity will eventually be attached to a CR Type, which is what users actually pick when creating a CR.
F
Fields to fill

What you enter when creating a Business Activity

Business Activity configuration screen
Business Activity*
A short code (usually 4 characters) that uniquely identifies this activity. Follow the SAP naming convention: {object short}{action digit}.Example: MAT1 = Material + Create
Description*
Plain-language description so configurators know what this activity does.Example: "Create Material"
Business Object Type*
The numeric BO type from the data model. Tells the system what kind of master data this activity operates on.Example: 194 (Material), 266 (Customer), 1000 (Cost Center)
Logical Action*
Pick from the SAP standard list. Tells the system what kind of operation this activity performs.Example: CREATE, CHANGE, DISPLAY, MASS

That's it. Just four fields. The simplicity is deliberate — a Business Activity is meant to be a tiny binding record, nothing more.

Without business activities, every CR Type would have to directly reference both a logical action and a BO type — and you'd repeat that combination everywhere. The Business Activity layer is the DRY principle for MDG: define the verb+object once, reuse it in 5 different CR types if needed.

Think of restaurant menus. The kitchen has standard cooking methods (grill, fry, bake, steam) — those are logical actions. The pantry has ingredients (chicken, fish, paneer) — those are BO types. The menu items (Grilled Chicken, Fried Fish, Steamed Paneer) are the business activities. The waiter (CR Type) just points to the menu item; the kitchen knows exactly what to do.

What is it

The Change Request Type

A Change Request Type is the reusable, configured blueprint that defines an entire kind of change request: what data model, what business activity (and therefore what operation), what workflow, what UI screen, and what edition (if any).

This is the most important configuration object for end users. When a user logs into MDG and clicks "Create Change Request," the first thing they see is a list of CR Types — and the one they pick determines everything that happens next.

Y
Why we use it

One template, many users

  • Reusability — you configure MAT01 once; 500 users can use it forever to create materials.
  • Standardization — every "Create Material" CR follows the same workflow, same approvers, same screen.
  • Separation of duties — Create CR Type (MAT01) goes through one workflow; Delete CR Type (MAT06) goes through a stricter workflow with more approvers.
  • Different UIs for different operations — MAT01's UI shows blank fields for entry; MAT03's UI shows read-only fields for display.

Each form type pre-configures which fields are shown, which validations apply, where the form is routed. CR Types are MDG's version of "ITR-1, ITR-2, ITR-3."

What is it

Single Record Processing

Single Record Processing means one Change Request handles exactly one master data record at a time. If you want to create one material, one CR is created. If you want to create five materials, you create five CRs — one per material.

This mode is activated by ticking the Single Object flag on the CR Type configuration. It is the default behavior for most standard CR Types like MAT01, BP01, and CUST1P.

Y
Why we use it

Why single-record mode exists

  • Clarity — one record, one approval. The approver sees exactly what is being changed, not a mixed bag.
  • Independent approvals — each record can have its own discussion, rejection, or revision without affecting others.
  • Granular audit trail — the CR number maps to one specific master record, making history easy to follow.
  • Better validations — entity-specific validations run cleanly without juggling multiple records.
  • Most business cases need only one record at a time — onboarding one new vendor, creating one new customer, updating one material's price.
F
Configuration fields involved

Where Single Record mode is switched on

You do not configure "Single Record Processing" as its own object — it is a behavior driven by a checkbox on the CR Type.

Single Record — related CR Type fields
Single Object*
Checkbox on the CR Type. When ticked → Single Record Processing is enabled.Example: MAT01 has this ticked → one material per CR.
Main Entity Type
The single entity this CR can create or change.Example: MATERIAL for MAT01, BUSINESS_PARTNER for BP01.
Business Activity
The activity bound to this CR Type — usually a single-record activity like MAT1, MAT2, BP1.
Workflow Template
Same workflow template (WS60800086) is used; the workflow simply carries one record's data.

If single-record processing didn't exist, you'd have to bundle changes into batches even for trivial one-off requests. Onboarding one new vendor would feel like running a mass-update project. The audit trail would also lose precision — instead of "CR 10004567 created vendor ACME Corp," you'd have "CR 10004567 created 1 of 15 vendors," which is harder to read in reports.

Think of ordering food at a counter. Single-record processing is like ordering one specific meal — you give a precise order, the kitchen prepares exactly that, and you get one tray back. It is the fast, focused way to handle ordinary requests. If you want five different meals for five different people, you'd place five separate orders rather than one giant mixed order — that's how single-record mode works in MDG.

L
What it links to

The Single Record concept's neighborhood

  • Up: the Change Request Type — the Single Object flag here is what switches this mode on.
  • Sideways: the Business Activity — usually a 1/2/3-suffix activity (MAT1 create, MAT2 change, MAT3 display).
  • Down: the Main Entity — the single root record (material, vendor, customer) the CR creates or modifies.
  • Opposite of: Multi-Object Processing (Section 08), which handles many records in one CR.

What is it

Multi-Object Processing

Multi-Object Processing (also called "mass processing") means one Change Request can carry multiple master data records at once. All records inside the CR are reviewed and approved together, then activated together.

This mode is activated by leaving the Single Object flag unticked on the CR Type. The Business Activity used typically has the suffix A (like MATA) and is bound to the logical action MASS.

Y
Why we use it

Why multi-object mode exists

  • Bulk efficiency — when you need to update 500 materials with a new unit of measure, opening 500 separate CRs is impractical.
  • Coordinated approval — sometimes records are related (a set of cost centers that must go live together) and need one combined yes/no decision.
  • Reduced approver load — one CR with 50 records = one notification, one review session, instead of 50 separate notifications.
  • Faster data migration — when loading legacy master data into MDG, multi-object CRs handle the volume.
  • Atomic activation — all records activate together at the same instant, avoiding inconsistent intermediate states.
F
Configuration fields involved

Where Multi-Object mode is switched on

Multi-Object — related CR Type fields
Single Object
Checkbox on the CR Type — leave un-ticked for multi-object mode.Example: a "mass material update" CR Type leaves this blank.
Business Activity
Use a mass-processing business activity (suffix A).Example: MATA = mass processing of material.
Logical Action
Bound through the business activity — typically MASS.Example: MATA → logical action MASS.
Main Entity Type
Still one entity type, but many instances of it inside the CR.Example: MATERIAL — but the CR carries records M1, M2, M3, M4…

If multi-object processing didn't exist, every record would need its own CR — making bulk updates and data migrations administratively impossible. Even worse: related records (like a parent cost center and its children) would activate at different times, creating inconsistent reporting states. Multi-object mode is what makes MDG practical at enterprise scale.

Think of grocery shopping. Single-record processing is like buying one item at a time — fine if you only need bread. Multi-object processing is like filling your entire weekly cart in one trip, going to checkout once, paying once. Far more efficient when you have a lot to buy. The cashier (= approver) reviews the whole cart together, and you either take it all or leave it all.

L
What it links to

Multi-Object's neighborhood

  • Up: the Change Request Type with Single Object un-ticked.
  • Sideways: the Business Activity — usually with suffix A (MATA) and logical action MASS.
  • Down: many instances of the same Main Entity packed into one CR.
  • Opposite of: Single Record Processing (Section 07).
  • Related: Editions (Section 12) — also a way to bundle multiple changes for coordinated activation.

What is it

The Change Request Step

A Change Request Step is a numbered checkpoint that the CR must pass through during its lifecycle. Each step is a state — "currently being processed by requester," "waiting for approver 1," "under final check," "completed," "in revision," etc.

Every CR has a current step at any moment in time. The workflow's job is to move the CR from one step to the next based on what users do (approve, reject, withdraw).

Y
Why we use it

Without steps, no workflow

  • Tracking — at any moment, the system (and users) know where the CR is in its journey.
  • Permissions — the current step determines who can see, edit, approve, or reject the CR.
  • UI behavior — different screens / buttons appear depending on the current step.
  • Routing — the next step is decided by the workflow's decision tables based on the current step.
F

Without steps, the CR would be a one-shot — submitted and immediately activated, with no approval, no review, no chance to catch errors. The whole concept of governance depends on having checkpoints, and the steps are those checkpoints.

Think of luggage on an international flight. Step 00 = check-in counter. Step 10 = security. Step 20 = customs. Step 90 = loaded onto the plane (final check). Step 99 = arrived at destination. Step 91 = sent back because it had a banned item. Each step has a clear owner, a clear set of actions, and a clear next-stop. The CR's steps work exactly the same.

What is it

Workflow Process Modeling

Workflow Process Modeling is the activity of designing the workflow that drives a Change Request — deciding the sequence of steps, who acts at each step, what conditions route a CR to one path or another, and what happens at the end.

Up to now we have been talking about configuration objects — CR Type, business activity, steps. Workflow Process Modeling is the activity of putting those objects together into a runnable journey. Think of it as the "blueprinting" phase of the workflow.

Y
Why we use it

Why workflow modeling is separate from running the workflow

  • Design before runtime — companies need to plan governance before they apply it. Workflow modeling is where the planning happens.
  • Reusability — once a workflow is modeled, it can be reused across many CR Types and many environments.
  • Visual review — modeling produces something stakeholders (business + IT) can review together before going live.
  • Adaptability — workflow logic changes over time. The modeling layer is where you change rules without writing code.
  • Compliance documentation — auditors want to see the documented workflow, not just code.
F
What you decide during modeling

The seven decisions of workflow modeling

Decisions you make when modeling a workflow
1. Workflow Template
Pick a starting template — usually a standard one delivered by SAP (see Section 14). Most modern projects pick WS60800086.
2. Step Sequence
Define the numbered steps (00, 10, 20, 90, 99) the CR will pass through.
3. Step Type per Step
Assign a step type (1, 2, 3, …) to each step — this controls which action buttons appear.
4. Agents (who acts at each step)
Assign users, roles, or organizational units to each step. Can be a fixed user, dynamic role, or rule-derived.
5. Routing Rules
Define rules (in the decision table) that decide where the CR goes after each action. Can be condition-based (if material price > 10000, route to senior approver).
6. Sequential or Parallel
Decide whether steps act one-after-another (sequential) or simultaneously (parallel).
7. Exit Strategy
What happens on approval (route to step 90 → activate), what happens on rejection (revision step 91 / withdraw / close).
F
Where modeling lives in SPRO

The configuration neighborhood

Workflow modeling spans several IMG nodes. You don't model in one screen — you model across several:

SPRO→MDG→General Settings→Process Modeling→Workflow

Under "Workflow" you'll find sub-items: Other MDG Workflows, Rule-Based Workflow, Define Change Request Steps for Workflow, Define Change Request Step Types, Configure Rule-Based Workflow.

If workflow process modeling didn't exist as a discipline, every project would jump straight to coding. There would be no design phase, no documentation, no way to discuss "should this need 1 or 2 approvers" before building. Projects would over-run, governance would be inconsistent, and changes would require developers every time. Modeling is what turns workflow into a configurable business asset instead of a coded technical asset.

Think of designing a city's metro line. Before laying any tracks, urban planners draw the stations, the connections, the express lines vs the local lines, the interchange points. Only after the design is reviewed and approved does construction begin. Workflow Process Modeling is the urban planning phase of MDG governance — you decide the stations (steps), the routes (sequential / parallel), and the rules (who can board at which station). The Rule-Based Workflow engine (Section 15) is then the actual train system that runs on these designed routes.

W
What is it

The two fundamental workflow patterns

Every workflow in MDG (and indeed in any business process) routes work in one of two patterns: Sequential — one approver acts, then the next, then the next; or Parallel — many approvers act simultaneously and the results converge before the workflow continues.

These are not separate workflow templates — they are patterns you configure within a workflow (typically the Rule-Based Workflow WS60800086). You decide pattern-per-step: step 10 might be sequential, step 20 might be parallel, depending on what your business needs.

Y
Why we need both

Different business needs, different patterns

  • Sequential is for hierarchical decisions — when each next approver depends on the previous one's verdict. A manager only reviews after the data steward has cleansed; a CFO only approves after the controller has validated.
  • Parallel is for independent reviews — when multiple stakeholders need to weigh in but don't depend on each other. Finance, Tax, and Legal can review at the same time; their opinions are independent.
  • Speed — parallel collapses many reviews into one elapsed-time window. A 4-approver parallel step finishes when the slowest approver is done, not when the chain has cycled through everyone.
  • Workload balancing — parallel routing means no single approver is a bottleneck.

Sequential workflow

Approvers act in order. Step 2 starts only after step 1 is complete. The CR walks down a chain like a relay race. Common patterns: 1-level (one approver), 2-level (two approvers in sequence), n-level (a chain as long as governance requires).

Parallel workflow

The CR fans out to many approvers at once. After all (or enough) of them act, the workflow converges based on rules:

  • All must approve — every parallel approver must say yes for the CR to proceed.
  • Any rejection re-routes — one "no" sends the CR back to the requester for revision.
  • Majority decides — threshold-based progression (e.g., 3 of 5 approvers must approve).
F
Configuration fields involved

Where the pattern is chosen

The pattern is configured in two places — at the CR Step level (which step is sequential vs parallel) and at the Rule-Based Workflow decision table level (where the routing rules live).

Fields that control routing pattern
Step Pattern (Sequential / Parallel)*
On the CR Step definition. Tells the workflow whether this step routes to one user or many.Example: Step 10 = Sequential (one approver), Step 20 = Parallel (Finance + Tax + Legal at once)
Agent / Assigned Users
For sequential: one user or role. For parallel: a list of users or roles.Example: Parallel step 20 → Finance Reviewer + Tax Reviewer + Legal Reviewer
Convergence Rule
For parallel steps only — how to decide when the step is "done." Common values: "All approve," "Any reject re-routes," "Majority approves."Example: ALL_APPROVE for compliance-critical changes; MAJORITY for everyday reviews
Timeout / Escalation
Optional. If any parallel approver doesn't act within X days, the step escalates to a backup user.
Next Step on Approval / Rejection
In the decision table — where the CR goes after this step finishes, based on the combined result.

If only sequential existed, parallel reviews would have to be faked by chaining "Finance → Tax → Legal" steps — which is slower (each waits for the previous) and gives the wrong audit trail (Tax appears to have reviewed "after" Finance, not "alongside"). If only parallel existed, hierarchical chains couldn't be enforced — your CFO might end up approving before the controller has even seen the request. Having both patterns lets MDG match the natural shape of every business process.

Sequential is like a school report card sign-off — first the class teacher signs, then the head of department, then the principal. Each signature waits for the previous.

Parallel is like a panel interview — three interviewers ask questions at the same time, all in the same room. At the end they pool their scores and decide together. Nobody waits for anyone else to "go first."

What is it

Business Add-Ins (BAdIs)

A Business Add-In (BAdI) is an SAP-defined hook point in the workflow where you can plug in custom ABAP code. SAP provides the interface (the method names, parameters, when it gets called); you provide the implementation (your own logic inside that method).

BAdIs are the "escape hatches" of MDG. When configuration alone cannot express what your business needs — like determining an approver from an external HR system, or calculating a special validation — BAdIs are where you write that logic.

Y
Why we use them

Why BAdIs exist alongside configuration

  • Beyond what config can do — some logic is too complex or too situational to express in decision tables.
  • Integration with external systems — call HR for approver lookup, send to a custom tracking tool, check an external compliance database.
  • Upgrade-safe customization — BAdI implementations live in your own namespace and survive SAP upgrades, unlike modifications to standard code.
  • Multiple implementations possible — different BAdI implementations can be active for different scenarios (per CR Type, per region, per company code).
  • Defined extension points — SAP guarantees these hooks will exist across releases.
F
The key BAdIs available

Important MDG workflow BAdIs you should know

Common MDG Workflow BAdIs
USMD_SSW_SYSTEM_METHOD_CALLER
Most-used BAdI. Called when a system service name is triggered in the rule-based workflow. Used for custom logic at specific steps — auto-fill fields, trigger external calls, custom validations.
USMD_SSW_PARA_FIND
Parameter finder. Used to determine values needed by the workflow at runtime — like deciding which approver group a CR should go to based on the data inside.
USMD_PROCESS_CR_AGENT_DEF
Agent determination. Used to dynamically find the user who should receive the CR at the next step — for example, looking up the requester's manager from HR.
USMD_RULE_SERVICE
Custom service for decision-table rules. Lets you implement a service name that appears in the decision tables and runs your code when that row's rule fires.
USMD_CREQUEST_MAINTAIN
CR maintenance hook. Called when CR data is being saved or updated — for custom validations, default-field filling, or cross-field logic.
USMD_CHECK_ENTITY
Entity-level checks. For custom validations on a specific master data entity (e.g., "material number must match pattern X").
USMD_NOTIFICATION_BADI
Notifications. Customize emails or alerts sent during the workflow (e.g., add custom recipients, change subject line, route to messaging tools).

How to access

Finding and implementing BAdIs

  1. T-Code: SE19 — create / edit BAdI implementations.
  2. T-Code: SE18 — view BAdI definitions (the interface SAP provides).
  3. IMG Path:
    MDG IMG→General Settings→Process Modeling→Workflow→Rule-Based Workflow→Business Add-Ins (BAdIs)

The IMG node typically links you to SE19 pre-filtered to the relevant BAdI definitions for MDG workflow.

Without BAdIs, every custom requirement would force you to modify SAP standard code — making the system unsafe to upgrade. Or worse, the requirement would be impossible to implement at all, forcing the business to change its processes to fit MDG instead of the other way around. BAdIs are the diplomatic channel between SAP standard and your company's reality.

Think of a USB port on a computer. The computer manufacturer provides the port (= BAdI definition) with specified shape, voltage, and protocol. You buy a USB device (= BAdI implementation) — a keyboard, a thumb drive, a microphone — and plug it in. The computer doesn't change; the port is the contract. You can swap devices without modifying the computer. SAP's BAdIs are USB ports for your custom workflow logic.

The Big Picture — End-to-End

Bringing every concept together in one full journey.

The configuration order (one-time setup)

Define the Data Model

Yesterday's class. e.g. MM for material — including the Main Entity Type (Material) and its Business Object Type (194).

Confirm the Logical Actions you'll use

Browse SAP's standard list (CREATE, CHANGE, etc.) — you only use, never create.

Create Business Activities

e.g. MAT1 = CREATE + 194 (Material). One per logical-action × BO-type combination.

Define Change Request Actions

Reuse SAP-delivered (Approve, Reject, etc.) and add custom actions if needed.

Define Step Types & assign Actions

Bundle actions into reusable presets (Step Type 2 = Approve + Reject in order).

Create the Change Request Type

e.g. MAT01 = MM data model + MAT1 activity + WS60800086 workflow + UI app/config.

Decide Single Record vs Multi-Object Mode

Tick or un-tick "Single Object" on the CR Type. Tick = one record per CR (most common). Un-tick = mass / multi-object processing.

Define Change Request Steps for the CR Type

e.g. for MAT01: step 00 (initial), step 10 (approval — step type 2), step 90 (final), step 99 (done).

Configure the Rule-Based Workflow Decision Tables

Wire each step to a user/role and define routing rules (sequential or parallel).

Configure UI Determination

Map (CR Type + step + role) to (UI app + UI configuration).

(Optional) Configure Editions

Group CRs into editions if this CR Type uses time-bound coordinated releases (mostly MDG-F).

(Optional) Implement BAdIs for custom logic

Plug in custom ABAP wherever standard configuration cannot express what you need — agent determination, external system calls, custom validations.

Concept-by-concept cheat sheet

Is: the permission-slip envelope around any master data change.
Why: ensures governance, audit, approval.
Linked to: CR Type, Workflow, Master data record.
Without it: no controlled master data updates possible.

Is: the metadata cover of a CR (CR Type, data model, workflow, UI references).
Why: tells the system what kind of CR this is.
Configured via: Change Request Type.
Without it: system can't interpret the CR.

Is: the verb — CREATE, CHANGE, DELETE, BLOCK, etc.
Why: standardized vocabulary across data models.
Linked to: Business Activity (which then attaches to CR Type).
Without it: system can't know what kind of operation a CR performs.

Is: the bridge — binds Logical Action + Business Object Type.
Why: reusable building block; defines verb+noun combinations.
Fields: Activity code, Description, BO Type, Logical Action.
Without it: every CR Type would re-declare action+object directly — chaos.

Is: the reusable template users pick when creating a CR.
Why: standardizes the configuration of similar CRs.
Fields: Code, Description, Data Model, Single-Object flag, Main Entity Type, Business Activity, Workflow Template, UI App, UI Config, Edition.
Without it: users would have to manually configure every CR.

Is: numbered checkpoints in the CR's lifecycle (00, 10, 20, …, 90, 99).
Why: tracks where the CR is; controls who can act on it.
Fields: CR Type, Step Number, Description, Step Type.
Without it: no workflow possible — CR can't progress.

Is: the buttons users click — Approve, Reject, Agree, etc.
Why: captures user decisions to advance the workflow.
Linked to: Step Type (which controls which actions appear at which step).
Without it: CR sits forever; no way to express decisions.

Is: reusable presets — "this group of buttons in this order."
Why: connects steps to actions without manual wiring at every step.
Fields: Step Type code, Description, Sequence, Action.
Without it: step-action wiring becomes a maintenance nightmare.

Is: the rule mapping (CR Type + step + role) → (UI app + UI config).
Why: different screens for different stages and roles.
Fields: CR Type, Step, Role, UI App, UI Config.
Without it: every user sees the same generic screen — confusing UX.

Is: grouped, time-bound bundles of CRs released together.
Why: synchronized release of related master data (mainly MDG-F).
Fields: Edition Type, Data Model, Description, Type Indicator.
Without it: uncoordinated activation of related reference data.

Is: the engine that routes the CR through its steps.
Why: automates handoffs; enforces rules; sequential or parallel routing.
Standard template: WS60800086 (rule-based).
Without it: CR is created but never moves anywhere.
