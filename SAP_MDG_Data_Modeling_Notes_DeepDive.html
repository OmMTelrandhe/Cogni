The chain of concepts — at a glance

Data modeling builds the "shape" of master data inside MDG. Everything else (Change Requests, Workflows, UIs) sits on top of this shape. The configuration flow:

Data Model→Entity Types→Attributes→Relationships→Hierarchies→BOTC→Key Prefixes→Authorization→Generate Structures→SMT Mappings→Activate

Once activated, the data model is ready to be referenced by Change Request Types, Business Activities, and Workflows in the Process Modeling layer.

W
What is it

The Data Model

A Data Model in MDG is the master blueprint that describes what kind of master data the system will govern. It defines the entities (what objects exist — material, vendor, customer, cost center), the attributes on those entities (fields like material number, description, plant), the relationships between entities (a material has plants, a plant has storage locations), and the hierarchies if any (cost-center hierarchy, account hierarchy).

Without a data model, MDG has nothing to govern. Process Modeling (CRs, workflows, UIs) only makes sense once a data model exists to point at.

Y
Why we need it

Why every MDG project starts here

  • Shape definition — MDG needs to know "what does a vendor record look like? what fields does it have?" before it can validate, route, or approve a vendor change.
  • Generation of structures — when activated, MDG generates dozens of ABAP database structures from the data model. These structures hold the staging and active data.
  • UI generation — the screens for Change Requests are partially generated from the data model. Fields the data model defines become fields on the CR screen.
  • Validation foundation — required fields, value tables, check tables — all anchored to the data model.
  • Extensibility — adding a new field to a standard SAP data model (like MM or BP) is how you customize MDG for your organization's needs.
F
The two deployment modes

Flex Mode vs Reuse Mode — the foundational choice

Every data model lives in one of two modes. This is the single most important architectural decision in MDG, and it affects everything downstream — where data is stored, how mappings work, whether ERP tables are used directly, and how the model behaves at runtime.

Flex Mode vs Reuse Mode — side by side
Reuse Mode
Active data is stored in existing ERP tables (MARA, BUT000, KNA1, LFA1). MDG only owns the staging area; once a CR is activated, data is written back to the standard ERP tables and read from there. The data model "reuses" the SAP-delivered ERP structures.Examples: MM (Material) and BP (Business Partner) — both Reuse Mode.
Flex Mode
Active data is stored in MDG-owned generated tables. There is no underlying ERP table — MDG creates its own tables from the data model. Useful when the data does not exist as standard master data in ECC/S4 (like custom reference data, finance-specific objects).Examples: 0G (Financials) — Flex Mode. Cost centers and accounts live in MDG-generated tables until activated.

This mode is selected via the Active Area field on the data model header (Section 02).

If a data model didn't exist, MDG would have nothing to govern. Change Request Types could not be created (they need to point to a data model). Entity types, attributes, relationships — all of which the CR screens, validations, and workflows depend on — would have no place to live. The data model is literally the floor on which the rest of MDG is built.

Think of a data model like the blueprint of a house. The blueprint says "this is a 3-bedroom house, with 2 bathrooms, a kitchen on the north side, the master bedroom on the east." Before any construction worker can lay a brick, the blueprint must exist. The bricks are like the data records, the rooms are entity types, the dimensions are attributes, and the corridors connecting rooms are relationships.

You don't keep redrawing the blueprint for every new house you build — you reuse it (Reuse Mode = standard SAP blueprint). Or, for a custom architectural style, you draw a brand-new blueprint (Flex Mode = MDG-only blueprint).

W
What is it

The Data Model Header

When you open a data model in transaction USMD_MODEL, the first thing you see is the header — a small set of fields that identify the model and define its overall behavior. These fields are set once and rarely changed; they govern everything beneath.

Y
Why these fields matter

The header decides the system's behavior at every layer

  • Identification — Data Model ID is referenced everywhere (CR Types, Business Activities, structures, mappings).
  • Storage routing — the Active Area field decides whether data goes to MDG-owned tables (Flex) or back to ERP tables (Reuse).
  • Transport & namespace — Package and Namespace control where generated objects land and whether they are customer or SAP owned.
  • Version control — Active Version flag tells the system which version is "live" (multiple inactive drafts can exist).
F

Indicator that this version of the model is currently active in the system. Only one version is active at a time; you can have multiple inactive draft versions for testing.

If header fields were not set correctly, the data model wouldn't activate, generated structures would land in the wrong package (breaking transports), and the Active Area choice would silently send data to the wrong place — either lost in Flex tables you didn't expect, or overwriting ERP tables you didn't want to touch. The header is small but every field has big consequences.

Think of the data model header like the title page of a book. It carries the book's name (Data Model ID), the author info (Package, Namespace), the edition number (Active Version), and the publisher's imprint that decides which warehouse it ships from (Active Area). Most readers skip the title page — but for the librarian (the SAP system), every line on that page determines where the book goes and how it is catalogued.

W
The simplest way to understand it

What the Storage & Use Type really decides

When you create an entity in MDG, you must tell the system three things about it:

  1. Does this entity have its own key, or does it borrow one?
  2. Does MDG need to store its data in a separate table, or just inside its parent?
  3. Is this entity something users create via CRs, or is it a list of pre-existing values they just pick from?

The Storage and Use Type (SU Type) is one single field that answers all three at once. There are four values — 1, 2, 3, 4 — and each is a different answer to those three questions.

The four SU Types — at a glance

Quick comparison: the 4 SU Types side by side
Type 1
"I am a king." Own key. Own storage table. Created via CR. Can be searched, viewed, edited as its own thing.MATERIAL, BP_HEADER, COSTCENTER
Type 2
"I am a duke under a king." Has its own key. Own storage table. But cannot exist without a Type 1 parent. Multiple Type 2s per Type 1 possible.MATERIAL_PLANT (under MATERIAL), BP_ADDRESS (under BP_HEADER)
Type 3
"I am a notebook page inside the king's folder." No own key. Stored inside the parent's table (or in a tightly bound sub-table). Just extends the parent with extra fields. Cannot be queried alone.MATERIAL_BASIC_DATA, MATERIAL_PURCHASING_DATA — additional groups of fields for MATERIAL
Type 4
"I am the phone book everyone reads." A reference / lookup list. Other entities point to it but no one creates instances of it via a CR. Has a key (because lookups need keys) and attributes (because lookups have descriptions), but is not the subject of governance.INDUSTRY_SECTOR, COUNTRY, ACCOUNT_GROUP

Type 1 — Independent Main Entity (the "King")

W
Detailed behavior

What makes Type 1 special

  • Has its own primary key — a single field or a composite that uniquely identifies one record (e.g., MATNR for MATERIAL).
  • Can be created independently — a CR can be opened just to create a MATERIAL, with nothing else.
  • Gets its own staging table — generated automatically (e.g. /1MD/MAT_OS01).
  • Has a BOTC assignment — must be linked to a Business Object Type Code (194 for material).
  • Is the "main entity" of CR Types — when you configure a CR Type, the Main Entity field is always a Type 1 entity.
  • Can use Internal or External key assignment — both are valid.

Tables generated for Type 1:

  • One staging table (header data) → e.g. /1MD/MAT_OS01
  • One key table → for tracking temporary keys and number range assignments
  • One active table (Flex mode only) — in Reuse mode, the active table is the standard ERP table (MARA)

Type 2 — Dependent Entity with Own Key (the "Duke")

W
Detailed behavior

What makes Type 2 different from Type 1

  • Has a composite key — the parent's key + its own key field(s). Example: MATERIAL_PLANT's key is (MATERIAL_NUMBER + PLANT_CODE).
  • Cannot exist alone — without a Type 1 parent, the Type 2 record is meaningless.
  • Has its own staging table — separate from the parent's.
  • Connected to parent via Qualifying relationship — the relationship "qualifies" the child by inheriting the parent's key.
  • Allows multiple instances per parent — one MATERIAL can have many MATERIAL_PLANT records (one per plant).
  • Typically uses External key assignment — the plant code is usually entered, not auto-generated. (Some cases use Internal, but External is most common.)

Tables generated for Type 2:

  • One staging table — contains BOTH the parent's key AND its own key + its attributes
  • One active table (Flex) — same structure
Parent & child analogy for Type 1 vs Type 2

Type 1 = a Person: has their own ID (passport number). Can exist alone in the world.

Type 2 = a Marriage: needs two people (the parent) to exist. Has its own ID (marriage certificate number), but the certificate only makes sense in the context of those people. You cannot have a "marriage" without the linked people.

Type 3 — Dependent, Sub-table or Extension (the "Notebook")

W
Detailed behavior

The most confusing type — clarified

Type 3 is the source of most confusion in MDG. Here's the clear truth:

  • Type 3 has NO key of its own. It does not get a separate primary key field.
  • It inherits its identity entirely from its parent. A Type 3 record exists only as "extra fields tied to one parent record."
  • 1:1 with its parent — exactly one Type 3 record per parent record. Never multiple.
  • Usually generated as columns inside the parent's table, or as a tightly linked sub-table that the parent reads/writes atomically.
  • Has attributes — yes! Type 3 carries fields. It's not just a flag; it's a group of attributes.
  • Connected to parent via Leading relationship with cardinality 1:1.

Why does Type 3 exist if it has no key? Because real-world master data has logical groupings. A MATERIAL has Basic Data, Purchasing Data, Sales Data, MRP Data — these are conceptually separate groups of fields, but they all belong to one material 1:1. Modeling them as separate Type 3 entities lets you:

  • Organize the data model cleanly into logical chunks.
  • Manage UI tabs (each Type 3 becomes a tab on the material screen).
  • Control authorization per data group ("only purchasing role can edit Purchasing Data").
  • Generate cleaner structures.

Common confusion: "Type 3 has no key, so how does the system know which parent record it belongs to?" Answer: through the relationship. Type 3 always has a Leading relationship to its parent (a Type 1 or Type 2). The parent's key is what locates the Type 3 record. There is no separate Type 3 key field because there's no need for one — the parent's key IS the address.

Type 4 — Reference / Check / Lookup Entity (the "Phone Book")

W
Detailed behavior

What Type 4 is, and what it is NOT

Type 4 is for reference data — lists of valid values that other entities point to. Industry sectors, countries, account groups, units of measure. Things you don't "create new ones of" via a CR; they pre-exist as part of customizing.

  • HAS a key — every lookup needs a code/ID to be referenced by (country = "IN", industry sector = "AUTOMOTIVE"). Type 4 absolutely has keys.
  • HAS attributes — typically a description, maybe a few classification fields. Yes, Type 4 carries attributes.
  • NOT created via CRs — the values are maintained as customizing or come from check tables in ERP. MDG just knows about them.
  • Other entities REFERENCE it — typically via a Referencing relationship or as a check-table link on an attribute.
  • Storage is usually external — Type 4 data often lives in standard ERP check tables (T005 for country, TVKO for sales orgs). MDG just maps to it.

Why have Type 4 in the data model at all?

  • So the data model knows the value is valid and exists.
  • So search helps and dropdowns can be generated.
  • So the relationship from another entity has a target to point to.
  • So authorization can be enforced (some Type 4 lists are restricted by role).

The tricky scenarios — where most people get confused

U
Tricky scenario #1

"You said Type 4 has attributes — but I see Type 4 attributes only when I create a Type 3 with a Qualifying relationship to the Type 4. How?"

This is the classic point of confusion. Here's the truth:

  • Type 4 has its OWN attributes — e.g. INDUSTRY_SECTOR has CODE, DESCRIPTION, INDUSTRY_GROUP. These are stored on the Type 4 lookup itself.
  • But those attributes are not directly visible inside a Type 1 entity. A MATERIAL doesn't have INDUSTRY_SECTOR's DESCRIPTION as one of its own fields.
  • The connection happens through a Type 3 sub-table — a small Type 3 entity belonging to the Type 1, which has a single field that references the Type 4 lookup's key (via a Qualifying or Referencing relationship).

Example:

  1. MATERIAL (Type 1) has many groups of data.
  2. One group is "Industry Classification" — modeled as a Type 3 entity called MATERIAL_INDUSTRY.
  3. MATERIAL_INDUSTRY has one attribute: INDUSTRY_SECTOR_CODE.
  4. That code is validated against the Type 4 entity INDUSTRY_SECTOR (which itself has CODE + DESCRIPTION + GROUP).
  5. On the UI, when the user picks an industry sector code in the material screen, MDG looks up the description from the Type 4 and displays it.

So in practice: Type 4's attributes (description, group) ARE accessed, but always through the Type 3 → Type 4 reference, not directly inside the Type 1. The Type 3 is the bridge.

U
Tricky scenario #2

"But isn't Type 3 only in a Leading relationship with its parent? How can it have a Qualifying relationship with Type 4?"

Type 3 has two different relationships simultaneously:

  1. Leading relationship (with its parent Type 1 or Type 2) — this is what makes Type 3 a "child of" the parent. The parent owns the Type 3.
  2. Qualifying or Referencing relationship (with a Type 4 lookup) — this is how the Type 3's attribute value gets validated against a list.

The Leading relationship is "I belong to this parent." The Qualifying relationship is "my value must be one of these lookup codes." Two different purposes, two different relationships.

Visualized:

MATERIAL (Type 1)
├── Leading relationship → MATERIAL_INDUSTRY (Type 3)
       └── Qualifying / Referencing relationship → INDUSTRY_SECTOR (Type 4)

So yes — Type 3 IS in a Leading relationship with its parent (always), AND can simultaneously have a Qualifying or Referencing relationship to a Type 4 lookup. These two relationships don't conflict because they go in different directions.

U
Tricky scenario #3

"Can Type 3 be the 'leading' side of a relationship — pointing to another entity?"

Yes, but with a specific meaning. When Type 3 is on the leading side of a relationship to a Type 4, it means: "this Type 3 sub-table's value must come from the Type 4 lookup." This is the Qualifying or Referencing direction we just discussed.

Type 3 is never the leading side of a relationship to a Type 1 or Type 2 — because Type 3 is dependent, it cannot "own" another entity. But it can point to a Type 4 lookup, which is a different kind of leading direction (a lookup reference, not an ownership).

U
Tricky scenario #4

"Type 1 with External key vs Internal key — which generates temp keys?"

This is where the Temporary Key Concept from Section 09 connects to SU Types:

Which entity types need temporary keys
Type 1 + External Key
No temp key needed. The user enters the key value (e.g. "M001") directly when creating the CR. The system uses this value throughout.
Type 1 + Internal Key
Temp key IS generated. The system assigns a temporary $-prefixed key (e.g. "$0000123") while the CR is in flight. At activation, the temp key is replaced by a real key drawn from the number range.
Type 2
Usually no temp key. Type 2's key is composite — parent key + its own key field(s). The parent's key may be a temp key (if Type 1 parent is Internal); the Type 2's own key field is usually external (entered by user).
Type 3
Never. Type 3 has no own key — it inherits the parent's key entirely. No temp key concept applies.
Type 4
Never. Type 4 is not created via CRs. Its keys exist beforehand as customizing/check-table values.

Summary rule: Only Type 1 with Internal Key Assignment needs a temporary key. Everything else either has the key entered by the user (External) or doesn't need one (Type 3, Type 4).

U
Tricky scenario #5

Valid SU Type ↔ Relationship Type combinations

Not all combinations of SU Type and Relationship Type are allowed. Here's the matrix MDG actually enforces:

From-Entity SU Type vs To-Entity SU Type — allowed relationship types
Type 1 → Type 1
Referencing (a material references a different material as base material). Cannot be Leading (two kings cannot own each other).
Type 1 → Type 2
Leading + Qualifying. The Type 1 owns the Type 2, AND the Type 2's key is qualified by (composed using) the Type 1's key.
Type 1 → Type 3
Leading (1:1). Type 1 owns the Type 3 sub-table. Cardinality must be 1:1 because Type 3 has no own key.
Type 1 → Type 4
Referencing. Type 1 looks up values from a Type 4 list. (Direct reference — usually used when the Type 4 value is a simple field on the Type 1.)
Type 2 → Type 3
Leading (1:1). Same logic as Type 1 → Type 3 — a Type 2 can also have its own Type 3 sub-tables.
Type 2 → Type 4
Referencing. Type 2 references a Type 4 lookup.
Type 3 → Type 4
Qualifying or Referencing. The Type 3 sub-table holds a value that must exist in the Type 4 lookup. Most common when the Type 3 represents a classification or grouping.
Type 4 → anything
Not allowed (Type 4 doesn't initiate relationships). Type 4 is always the target of relationships, never the source. Lookups don't "own" or "point to" other entities; other entities point to them.

The mental model — when in doubt, ask these questions

U
Decision tree

How to pick the right SU Type for a new entity

  1. Is this entity something users create via a CR?
    • NO → it's a lookup/reference list → Type 4.
    • YES → go to step 2.
  2. Can it exist on its own without a parent?
    • YES → Type 1. Has its own key, its own life.
    • NO → go to step 3.
  3. Can there be multiple of these per parent?
    • YES (1:N) → Type 2. Needs its own key field(s) to distinguish siblings.
    • NO (1:1) → Type 3. Just a group of extra fields for the parent.
One unified analogy — the customer at a bank
  • Customer = Type 1 — has unique customer ID. Created when account opens. Stands alone.
  • Bank Accounts = Type 2 — customer can have many (savings, current, fixed deposit). Each has its own account number, but cannot exist without the customer.
  • KYC Details = Type 3 — one block of KYC fields per customer (1:1). No separate "KYC ID" — just extra fields hung onto the customer.
  • Account Type List (Savings/Current/FD) = Type 4 — a fixed list of account types maintained by the bank. Customers don't create "new account types" via a form; they just pick one.
L
What it all connects to

SU Type's full neighborhood

  • Determines: key behavior (temp keys for Type 1 Internal only), table generation (full set vs sub-table vs none), CR creatability (Type 1 only), BOTC linkage (Type 1 only).
  • Determines allowed: relationship types (Leading from Type 1/2, Qualifying for Type 2 keys, Referencing to Type 4), cardinalities (1:1 mandatory for Type 3).
  • Affects: UI layout (Type 3 = tabs, Type 2 = sub-screens, Type 4 = dropdowns), authorization (per-entity flag), search relevance.
  • Cannot be changed after activation — once a data model with a chosen SU Type is activated and used in CRs, changing the type means destroying and recreating the entity. Choose carefully.

W
What is it

The Attribute

An Attribute is a single field on an entity type — a piece of data the entity carries. Material has attributes like description, base unit of measure, material group, weight. Business Partner has name, address, tax classification, payment terms.

Each attribute is backed by an ABAP Data Dictionary data element, which decides its data type (CHAR, NUMC, DATS), length, and value range.

Y
Why we need attributes

Why each attribute is configured deliberately

  • They are the actual data — without attributes, an entity is an empty shell with only a key.
  • They drive UI fields — fields on the CR screen are generated from the attributes.
  • They drive validation — required flag, value range (via data element domain), and check tables all operate at the attribute level.
  • They drive search — search-relevant attributes become searchable filters in the MDG UI.
  • They drive structure generation — each attribute becomes a column in the generated structures and tables.
F

Attribute fields

Human-readable label shown in the UI.
Data Element*
The ABAP DDIC data element that gives this attribute its data type, length, search help, and domain (value range).Example: MAKTX for material description, INDUSTRY for industry sector

U
Internal concept — Check Tables & Value Tables

How attribute values are validated

An attribute can be linked to a check table via its data element's domain. When the user types a value, MDG validates it against the check table — like "this material group must exist in T023."

  • Value Table — defined at the domain level. Just a recommendation; not enforced.
  • Check Table — defined at the field/data-element level. Strictly enforced at runtime.
  • "No Existence Check" flag — bypasses the check table validation if you want free text.

This is why getting the data element right is crucial — the entire validation behavior of an attribute is inherited from it.

If attributes didn't exist, an entity would just be a name with a key — no actual data. If their validation flags (Required, Existence Check) didn't exist, users could enter blank or junk values, polluting master data. The validation flags on the attribute are what turn the data model from a passive shape into an active gatekeeper.

Think of attributes like the fields on a job application form. "Full Name" is required (Required flag), "Country" must be picked from a dropdown (check table), "LinkedIn URL" is optional and free-text (No Existence Check). The form designer decides which fields exist, which are mandatory, which validate against a list. MDG's attribute configuration is the form designer's choices, written in SAP customizing.

  • Becomes: a column in the generated staging table and in the field-property structure.
  • Influences: the CR UI screen layout (via Floorplan Manager config).

What is it

The Relationship

A Relationship is a configured link between two entity types in the same data model. It tells MDG how the entities relate at runtime — which one is the parent, which is the child, how many of one can exist for the other.

Relationships are how you build composite master data structures: a material has many plants, each plant has many storage locations; a business partner has one or more addresses, one or more bank accounts.

Y
Why we need relationships

Why relationships are not "optional"

  • Data integrity — MDG can prevent orphaned records (e.g., a plant entry without a parent material).
  • UI navigation — the relationship structure drives how the user navigates tabs and sub-screens in a CR ("Open material → click plants tab → see linked plants").
  • Cascading actions — when a parent is deleted, dependent children behave according to the relationship type.
  • Generated structures — relationships drive how the staging tables are linked (foreign-key columns).
F
The Relationship Type — Leading / Qualifying / Referencing

The three relationship types

Relationship types explained
Leading
Strong "owning" relationship. The "to" entity exists only because of the "from" entity. Deleting the parent deletes the children.Example: MATERIAL → MATERIAL_PLANT. If MATERIAL is deleted, the linked PLANT data is meaningless.
Qualifying
The "to" entity's key is composed of the "from" entity's key. The child cannot be uniquely identified without the parent.Example: MATERIAL + PLANT together identify a MATERIAL_PLANT record — PLANT alone is ambiguous.
Referencing
Weak link — pointer only. The "from" entity points to a "to" entity which lives independently. Used for lookup-style links.Example: BP_HEADER → INDUSTRY_SECTOR. The industry sector exists on its own; the BP just references it.
F
Relationship — full field reference

Fields you fill when creating a relationship

Relationship fields
Relationship Name*
Unique name identifying this relationship.
From Entity Type*
The "parent" or "source" entity in this relationship.Example: MATERIAL
To Entity Type*
The "child" or "target" entity.Example: MATERIAL_PLANT
Relationship Type*
Leading / Qualifying / Referencing as explained above.
Cardinality*
How many of "to" per "from." Standard SAP notation:
· 1:1 — exactly one
· 1:N — many
· 1:CN — zero or many (optional)
· 1:C — zero or oneExample: MATERIAL → MATERIAL_PLANT is 1:CN (a material may have zero or many plants)

If relationships didn't exist, entities would be isolated islands. You could create a MATERIAL_PLANT record pointing to nothing. The UI couldn't navigate from material to its plants because the link wouldn't be defined. Generated structures would have no foreign keys. Relationships are the connective tissue of the data model.

Think of a family tree. A parent-child link is Leading (the child wouldn't exist without the parent). The relationship between siblings sharing a parent is Qualifying (you need to know "child of which parent" to uniquely identify them). A person's reference to their country of citizenship is Referencing — the country exists independently, the person just points to it. Cardinality is "how many" — one country per person (1:1), many children per parent (1:N).

L
What it links to

Relationship's neighborhood

  • Connects: two Entity Types within the same Data Model.
  • Drives: foreign-key columns in generated tables, UI tab/subscreen navigation, cascading delete behavior.
  • Influenced by: the SU Types of the two entities (Type 2 entities almost always have a Qualifying relationship to their parent Type 1).

What is it

The Hierarchy

A Hierarchy in MDG is a tree-shaped organization of an entity type's records — like a cost center hierarchy where "Asia" has child "India" which has child "Mumbai branch." It is a special kind of relationship: a parent-child link where any node of the entity can have children of the same entity type.

Hierarchies are most heavily used in MDG-F (Financials) for cost centers, profit centers, account hierarchies, and consolidation structures.

Y
Why hierarchies are first-class citizens

Why MDG treats hierarchies specially

  • Reporting — financial reports aggregate at hierarchy levels. The hierarchy is the report structure.
  • Authorization — you can grant rights at hierarchy nodes ("approve everything under Asia").
  • Versioning — hierarchies change over fiscal periods. MDG supports edition-dependent hierarchies for clean fiscal-year cutovers.
  • Display — MDG ships special UI widgets for tree views that only work if the entity is configured as hierarchical.
F
Configuration

How a hierarchy is enabled

A hierarchy is enabled at two levels: a flag on the entity type, and an optional configuration of "edition-dependent" behavior.

Hierarchy-related fields
Hierarchy Capability (on Entity Type)*
Flag on the entity type. Tells MDG this entity is hierarchical — generates the parent-child storage tables, exposes hierarchy navigation in the UI.
Hierarchy Type
The kind of hierarchy:
· Simple — one parent per node, no versioning.
· Version-dependent / Edition-dependent — different parent depending on the time period or edition.
Edition-Dependent Hierarchy
Checkbox. If ticked, the hierarchy structure is bound to an Edition (release-bundle). Different editions can have different hierarchy snapshots — essential for "this is our cost-center tree for FY2026 Q1, this is the new one for Q2."

If hierarchies weren't a built-in concept, you'd have to fake them by making each entity self-reference itself with a normal relationship — possible, but you'd lose tree-UI widgets, versioning, edition support, and hierarchy-based authorization. MDG's first-class hierarchy support is what makes Finance master data manageable.

A hierarchy is like the org chart of a company. The CEO sits on top, VPs underneath, managers under each VP, individual contributors at the bottom. Two important features:

  1. The chart can be re-organized — last quarter's chart is different from this quarter's. That's edition-dependence.
  2. Authorization works "everyone in Engineering can approve" — you don't list each engineer, you point to the Engineering node. That's node-based authorization.

MDG's hierarchy mechanics power exactly these scenarios for cost centers, profit centers, and accounts.

The Business Object Type Code

A Business Object Type Code (BOTC) is a numeric or short-string code that SAP uses across all its products to identify a specific kind of master data object — material is 194, business partner is 147, supplier is 266, customer is KNA1 in older systems and BUS1006 in newer ones.

What is it

The Temporary Key Concept

When a user creates a CR for a record with Internal Key Assignment (system-generated key), there is a problem: the real key shouldn't be issued until the CR is approved (so we don't waste ID numbers on rejected CRs). But the user needs some identifier on screen — they can't fill the form for "the next material" if there's no field to show what they're creating.

The solution: a Temporary Key. While the CR is in flight, MDG assigns a temporary ID prefixed with $ (default). When the CR is activated, the temporary key is replaced with the real one drawn from the number range.

Y
Why temporary keys exist

Why we can't just issue real keys upfront

  • Reject-safe — if the CR is rejected, no real material number is wasted.
  • Multiple draft CRs — users can have several drafts open simultaneously, each needing a unique identifier without conflicting.
  • Predictable behavior in mass processing — when 50 materials are being created in one CR, each gets a distinct temporary key inside MDG; real keys are assigned at activation.
  • Audit clarity — temporary keys are visibly distinguishable (the $ prefix) so it's clear in logs which records are still in flight.

U
Internal concept — Number Range Objects

The link to SNRO number ranges

The real key (assigned at activation) comes from a Number Range Object defined in transaction SNRO. The data model's entity-type configuration includes the Number Range Object name. At activation, MDG calls the number range and replaces the temporary key.

  • For Material (MM) — real number ranges are defined in standard SAP, attached via account groups.
  • For BP — number ranges per BP grouping; the data model points to BU_PARTNER's number range setup.
  • For custom models — you create a Z number range object in SNRO and assign it to your entity type.

If temporary keys didn't exist, users would have to type a "guess" key for every create CR (which defeats the purpose of internal assignment), or MDG would issue real keys that get wasted on rejection. Either option breaks the audit trail and the number-range hygiene. The temporary key concept is what makes "create with internal key" workable inside a workflow-driven world.

Think of a hospital admitting a new patient who hasn't received their hospital ID yet. The nurse writes "Pt-A" on the wristband (temporary key) and starts filling in forms. When the patient is officially admitted (CR activated), they get a real hospital ID like "HPT-2026-0457" (real key from number range). The temporary ID is short-lived; the real one stays forever in the medical record.

Authorization Relevance

Authorization Relevance is the data-model-level setting that tells MDG "for this entity (or specific attribute), check the user's authorizations before allowing the action." Without this flag, MDG skips the auth check on that entity, effectively letting any logged-in user touch the data.

Once an entity is marked authorization-relevant, MDG uses three standard authorization objects to enforce access at runtime: R_USMD_MOD, R_USMD_VAL, and R_USMD_CRQ.

Y
Why we mark entities authorization-relevant

Why authorization is data-driven, not just role-driven

  • Granular control — you can permit a user to change MATERIAL but not COSTING. The flag is per-entity (and per-attribute if needed).
  • Compliance — sensitive data (bank details, salary fields) can be locked to specific roles even within the same entity.
  • Approver scoping — only users authorized for the entity can see the CR's content for approval.
  • Decouples role design from data-model design — the data model marks WHAT is sensitive; the role marks WHO can do what to it.

Structure Generation

Once your data model is defined (entities, attributes, relationships, hierarchies), it is still just metadata. W
What is it

The Two Storage Areas

MDG separates master data into two zones — the Staging Area (where in-flight CR data lives) and the Active Area (where approved, live master data lives). Every record flows from Staging to Active when its CR is activated.

Staging vs Active — side by side
Staging Area
The "in-flight" zone. Holds CR data while it is being drafted, reviewed, approved. Always MDG-owned tables (generated from the data model). Multiple versions of a record can coexist in staging (different CRs proposing different changes).Example: tables like /1MD/MAT_OS01 (MDG-generated material staging)
Active Area
The "live" zone. Where approved master data permanently resides. Where it lives depends on the data model's mode:
· Reuse Mode — Active Area = standard ERP tables (MARA, BUT000, KNA1)
· Flex Mode — Active Area = MDG-generated active tables (separate from staging)
Y
Why split staging from active

Why MDG isolates draft data from live data

  • Safety — rejected/withdrawn CRs never pollute live data. The staging area is a sandbox.
  • Concurrency — multiple CRs can propose conflicting changes to the same active record without trampling each other.
  • Audit — staging holds the full history of proposed changes, including those that were rejected.
  • Validation room — staging is where validations run before commit. If a validation fails, the data is contained.
  • Comparison — MDG can show "before vs after" by comparing staging to active during the approval screen.
F
How a record flows between them

The Staging → Active journey

  1. CR created — user enters data; MDG writes it to staging.
  2. CR in review — approvers see staging data; active data is untouched.
  3. Approval granted — CR reaches step 90 (final check).
  4. Activation — MDG copies the staging data to the active area. Real keys replace temporary ones. Number ranges are consumed.
  5. Staging cleanup — the staging entries are marked as processed (kept for audit but no longer active).
U
Internal concept — Reuse Mode's special behavior

Why Reuse Mode is uniquely powerful

In Reuse Mode, the Active Area is the standard ERP tables. This means:

  • Zero data duplication — once a material is activated, it lives in MARA, just like materials created via MM01 directly.
  • Existing reports work unchanged — your customer's reports, transactions, BAPIs continue reading from MARA without knowing MDG exists.
  • Coexistence — MDG-governed and non-MDG-governed materials sit side by side in MARA.
  • SMT mapping is critical — because the staging area is MDG-generated and the active area is ERP-standard, you need to map fields between them (Section 13).

In Flex Mode, the active area is a separate MDG-generated set of tables. Mapping is simpler (or unnecessary) because both staging and active share the same schema, but you lose the "everything just works with ERP" advantage.

If MDG didn't separate staging from active, every keystroke in a CR would immediately hit live data. Rejected CRs would have already polluted the database. The very idea of "approval before commit" would be impossible. The staging/active split is the architectural foundation that makes governance possible.

Think of publishing an article on a blog. While you write, the post lives in "Drafts" (staging). Multiple drafts can exist. Reviewers can read your draft, comment, ask for revisions. When you click "Publish," the draft is copied to "Live" (active) where the world sees it. Old drafts stay around for revision history, but they no longer affect what's public.

MDG works exactly this way. CR submission = draft. Approval workflow = peer review. Activation = publish.

L
What it links to

Active/Staging's neighborhood

  • Configured via: Active Area field on the data model header.
  • Storage: Staging always in MDG-owned tables; Active in ERP tables (Reuse) or MDG-owned tables (Flex).
  • Mapped via: Service Mapping Tool (SMT) — see next section.
  • Triggered at: CR step 90 (final check & activation).

What is it

Service Mapping Tool (SMT)

The Service Mapping Tool (SMT) is the configuration framework where you tell MDG: "this field in my staging structure corresponds to that field in the ERP table." It is the dictionary that lets MDG read from and write to ERP tables without code.

SMT is used primarily in Reuse Mode data models (MM, BP). In pure Flex Mode, no external mapping is needed because the active and staging are both MDG-owned.

Y
Why SMT exists

Why mapping is not optional in Reuse Mode

  • Staging and active have different shapes — MDG's staging structures (generated) don't necessarily match the field names or layouts of ERP tables (MARA, BUT000).
  • Field name divergence — MDG might call a field MATERIAL_DESC while MARA calls it MAKTX. SMT translates the names.
  • Activation needs translation — when a CR activates, MDG writes to the ERP table; the mapping tells it which ERP field to populate from which staging field.
  • Reading existing data — when a user starts a CR to change an existing material, MDG reads MARA via the mapping to pre-fill the screen.
  • Custom extensions — when you add a custom Z field to MARA, you map it through SMT to your custom MDG attribute.
F
Two flavors of mapping

SMT Mapping to Active Area vs to Staging Area

SMT mapping types
SMT Mapping to Active Area
Used at CR activation. Source: MDG staging structure. Target: ERP active table. Tells MDG which ERP fields to write to during activation.Example: MDG MATERIAL.MATERIAL_DESC → MARA-MAKTX
SMT Mapping to Staging Area
Used at CR creation (for change CRs). Source: ERP active table. Target: MDG staging structure. Tells MDG how to read the current ERP record and load it into staging for the user to edit.Example: MARA-MTART → MDG MATERIAL.MATERIAL_TYPE

Both mappings are needed for a full round-trip: read from active → edit in staging → write to active.

U
Internal concept — CVI for Business Partner

CVI (Customer-Vendor Integration)

For the BP data model specifically, there is an additional layer called CVI — Customer-Vendor Integration. CVI synchronizes BP records with the classic customer (KNA1) and vendor (LFA1) tables.

  • A BP marked as "Customer" role automatically gets a corresponding KNA1 entry created via CVI.
  • A BP marked as "Supplier" role syncs with LFA1.
  • CVI is configured separately but works alongside SMT — SMT handles MDG↔BUT000, CVI handles BUT000↔KNA1/LFA1.

For S/4HANA, BP is the only allowed master for customers and suppliers; KNA1/LFA1 are still maintained by CVI for backwards compatibility.

F
Configuration

Accessing SMT

T-code→MDG_BS_SMT · or · SPRO→MDG→General Settings→Data Modeling→Create and Edit Mappings

SMT configurations are only applicable in Reuse Mode. In pure Flex Mode, SMT is not used because there is no separate active store to map to.

Without SMT in Reuse Mode, MDG could not write activated CRs back to ERP tables. The active area would always be empty. Every read of existing ERP records (for "change material" CRs) would fail. SMT is the bridge that makes Reuse Mode practical — it lets MDG live alongside ERP without duplicating data.

Think of SMT like a translator at a UN summit. The MDG speaker (staging) and the ERP speaker (active) talk different languages. The translator (SMT) listens to one side, converts the meaning, and speaks it to the other. Both sides retain their native vocabulary; only the translator knows the dictionary.

Without the translator, the two sides could meet but never exchange information meaningfully. With the translator, they collaborate seamlessly — even though their internal field names and structures differ.

L
What it links to

SMT's neighborhood

  • Connects: MDG staging structures ↔ ERP active tables.
  • Triggered during: CR creation (read), CR activation (write).
  • Only used in: Reuse Mode data models (MM, BP).
  • Complemented by: CVI for BP (BUT000 ↔ KNA1/LFA1).
  • Customized via: SMT extension framework (see Section 14).

What is it

Extending SMT Mappings

When your business needs a field that SAP doesn't ship — say, a "Hazard Class Code" on materials or a "Loyalty Tier" on customers — you don't modify standard data models. You extend them. Extending means: add your custom field to the ERP table (via append structure), add the matching attribute to the MDG data model, regenerate structures, and finally extend the SMT mapping so the new field round-trips correctly.

This is the most common customization activity in MDG projects.

Y
Why we extend instead of modify

Why extension is the SAP-approved approach

  • Upgrade safety — SAP can deliver new versions of MM, BP, 0G; your extensions stay intact because they live in customer namespace.
  • Transport-friendly — extensions are tracked separately, easy to move across landscapes.
  • Reusability — extending standard models means you keep all the SAP-delivered behavior (validations, BAdIs, UI configs) and just add fields on top.
  • Audit clarity — extensions are clearly labeled as customer-specific.
F
The extension sequence

The five-step procedure for adding a custom field

  1. Extend the ERP Table — use append structure (SE11) to add the field to MARA or BUT000 in customer namespace (ZZ-prefix).
  2. Extend the Data Model — in USMD_MODEL, add the attribute to the appropriate MDG entity, point it to the same data element.
  3. Regenerate Structures — re-run "Generate Data Model-Specific Structures" so the new attribute appears in the generated staging structure and field-property structure.
  4. Extend SMT Mapping — open SMT, add a row mapping your new ERP field to the new MDG staging attribute (for both directions — to-active and to-staging).
  5. Test — create a CR for the relevant business activity, confirm the field appears on the UI, save, approve, verify activation writes correctly to the ERP table.

Skipping any step breaks the chain. Forgetting step 3 (regenerate) is the most common cause of "the field doesn't appear on the screen."

F
Fields involved in the extension

What you fill at each step

SMT mapping extension — fields
Source Field
The field name in the source structure (ERP for to-staging, MDG for to-active).
Target Field
The field name in the target structure.
Transformation Rule
Optional. If the source and target have different formats (e.g., different units, different code values), specify a transformation. Otherwise, "1:1 copy" is the default.
BAdI Implementation
For complex transformations (lookup tables, derivation logic), assign a BAdI implementation that runs at mapping time.

If you can't extend, you can't customize. Every business has unique fields they need — without extension, MDG would be a take-it-or-leave-it product, useless for any company with non-trivial needs. Extension is what makes MDG practical in the real world.

Think of buying a pre-built house. The architecture (standard data model) is fixed. But you can knock down a wall and add a study (extend the ERP table), update the floor plan to reflect it (extend the data model), redo the wiring diagram (regenerate structures), and tell the housekeeper where the new room is (extend SMT mapping) — without rebuilding the house from scratch. Extension is the renovation discipline of MDG.

L
What it links to

Extension's neighborhood

  • Touches: ERP table (append), Data Model (new attribute), generated structures, SMT Mapping.
  • Often complements: BAdI implementations for custom logic.
  • Critical for: every customer-specific MDG implementation.

What you need to know

The three big standard models — at a glance

SAP ships three flagship data models with MDG. They cover the bulk of real-world master data governance needs:

  • MM — Material Master (Reuse Mode)
  • BP — Business Partner, including Customer / Supplier roles (Reuse Mode)
  • 0G — Financials, including Cost Center, Profit Center, Account, Hierarchies (Flex Mode)

Each has its own personality — its own root entity, its own key tables, its own quirks. Understanding them is essential.

15.1 · MM — Material Master

F
Anatomy of the MM data model

MM at a glance

MM data model characteristics
Root Entity
MATERIAL — represents one material master record (key: MATNR / MATERIAL_NUMBER).
Storage Mode
Reuse Mode. Active data lives in the standard ERP material tables — your existing reports keep working unchanged.
Key Active Tables
MARA (general data) · MARC (plant data) · MBEW (valuation/accounting). MDG governs creates/changes; activation writes to these tables.
BOTC
194 — Material Business Object Type Code.
CR Types Delivered
MAT01 (Create) · MAT02 (Change) · MAT03 (Display) · MATA (Mass).
Extension BAdI
MDG_BS_MAT_API_SEGMENTS_EXT — the standard extension hook for custom field logic in materials.

15.2 · BP — Business Partner

F
Anatomy of the BP data model

BP at a glance

The BP data model unifies what used to be three separate masters in classic ERP — Business Partner, Customer, Supplier. In S/4HANA, BP is the only entry point; KNA1 and LFA1 are maintained automatically via CVI.

BP data model characteristics
Root Entity
BP_HEADER — represents one business partner record (key: BU_PARTNER / BP_ID).
Storage Mode
Reuse Mode. Active data in BUT000 and related BP tables.
Key Active Tables
BUT000 (BP header) · BUT020 (addresses) · BUT100 (roles). Plus, via CVI: KNA1 (customer view) and LFA1 (vendor view).
BOTC
147 — Business Partner BOTC. 266 for Supplier role. Also BUS1006 for SOA contexts.
CR Types Delivered
BP01, CUST1P, SUPP1P, BP02 and their family for change/display.
Integration Layer
CVI — Customer-Vendor Integration. Syncs BP records to/from classic customer (KNA1) and vendor (LFA1) tables.
GenIL (Generic Interaction Layer) — used by older BP UI front-ends; in modern MDG, GenIL provides the runtime data access for some BP scenarios.
U
Internal concept — BP roles

One record, many roles

A single BP record can simultaneously play multiple roles: General BP, Customer, Supplier, Employee, Contact Person. The role assignment is stored in BUT100. Each role can bring its own set of fields and screens. MDG's BP data model is designed to handle this multi-role nature natively — you don't create separate vendor and customer masters, you create one BP and assign roles.

15.3 · 0G — Financials

F
Anatomy of the 0G data model

0G at a glance

0G is the financial reference-data model — cost centers, profit centers, accounts, account hierarchies, and consolidation structures. It is uniquely Flex Mode, edition-driven, and hierarchy-heavy.

0G data model characteristics
Storage Mode
Flex Mode. Active data lives in MDG-owned tables, not directly in ERP CO/FI tables. Activation pushes data out via the Master Data Mass Generation (MDMGX) tool when bringing changes into the operational system.
Edition-Driven
Everything in 0G is bound to Editions. A cost-center hierarchy for Q1 is a different edition from Q2. This enables planned future-dated changes that activate together at a fiscal cutover.
Key Entities
CCTR (Cost Center) · CELEM (Cost Element) · PCTR (Profit Center) · ACCOUNT (G/L Account) · plus hierarchy entities for each.
Master Data Load Tool
MDMGX — used to bulk-load initial financial master data into 0G, and to push approved 0G changes downstream to ERP CO/FI when needed.
U
Why 0G is Flex Mode

The architectural reason

Financial reference data needs planning capability: "we want this new cost center structure to go live on April 1 next year." If 0G were Reuse Mode, every change would immediately update CO/FI tables — no planning room. By being Flex Mode + Edition-driven, 0G lets you:

  • Draft a new cost-center hierarchy for next quarter while the current one stays untouched.
  • Have multiple competing draft structures (different planning scenarios) coexist.
  • Activate the entire edition (potentially 100+ CRs) atomically on the target date.
  • Report on "what does our cost-center tree look like as of June 1?" — historical and future views.

Without standard data models, every MDG project would start from a blank canvas — defining materials, vendors, cost centers from scratch. SAP's pre-delivered MM, BP, and 0G models save months of work and bring proven integration with ERP, CVI, DRF, and BRF+. They are the starting point of almost every real MDG implementation.

Think of these three models like the three flagship apps that come pre-installed on a new phone: Phone (MM — used by everyone, fundamental), Contacts (BP — central, identity-defining), Calendar (0G — time and planning oriented). You'll add your own apps over time (custom data models), but these three are the bedrock that supports the everyday experience.

L
What they link to

The standard models' neighborhood

  • MM ↔ ERP MARA/MARC/MBEW (Reuse), CR Types MAT01/02/03/A, BOTC 194.
  • BP ↔ BUT000/BUT020/BUT100 + CVI to KNA1/LFA1, CR Types BP01/CUST1P/SUPP1P, BOTCs 147/266.
  • 0G ↔ MDG-owned tables + Editions + MDMGX, hierarchy-heavy.
  • All three are extended via the procedure in Section 14 for customer-specific fields.

The Big Picture · Full Chain

Bringing every Data Modeling concept together in one end-to-end view.

The complete data modeling chain

Here is how all 15 concepts you've just learned connect — in the order you'd actually configure them on a real project:

1. Create Data Model (USMD_MODEL): set header fields (ID, description, Active Area, Package).
2. Define Entity Types: pick SU Type (1, 2, 3, 4), key assignment, hierarchy capability.
3. Add Attributes: data elements, key flags, required flags, search relevance.
4. Define Relationships: Leading / Qualifying / Referencing + cardinality.
5. Enable Hierarchies if needed (edition-dependent or simple).
6. Define BOTCs if creating custom data models; reuse 194/147/266 for standard.
7. Map Entity → BOTC: tell MDG which entity is the primary for each BOTC.
8. Configure Temp Key Prefixes: $-prefix for in-flight internal keys.
9. Mark Authorization Relevance: flag entities/attributes, wire to R_USMD_* objects.
10. Generate Structures: produce mandatory + optional ABAP DDIC artifacts.
11. Set up SMT Mappings (Reuse Mode only): staging ↔ active translations.
12. Extend as needed: append fields → re-generate → extend SMT.
13. Activate Data Model: model goes live; CR Types can now reference it.
14. Verify: SE11, MDG_DATA_MODEL, USMD_ADJUST_STAGING.

How Data Modeling feeds into Process Modeling

Once a data model is activated, it becomes the foundation that Process Modeling reads:

  • A Change Request Type picks a Data Model and a Main Entity (a Type 1 entity from the data model).
  • A Business Activity references a BOTC, which resolves via the entity-BOTC mapping to a specific entity.
  • The UI Configuration for the CR uses the field-property structure generated by the data model.
  • The Workflow reads authorization flags on entities to determine who can act on which CR.
  • Activation of the CR uses the SMT Mapping (in Reuse Mode) to write back to ERP tables.

Mental model — the layered system

Bottom (foundation): Data Model · Entities · Attributes · Relationships · Hierarchies
Storage: Staging Area + Active Area (Reuse OR Flex)
Integration: BOTC · Entity↔BOTC · SMT Mapping · CVI · GenIL
Generation: Structures · Tables · Field Properties · Search · DRF
Security: Authorization (R_USMD_MOD · R_USMD_VAL · R_USMD_CRQ)
Top (process): Change Request Type · Business Activity · Workflow · UI

Data Modeling is the foundation; Process Modeling is the building.
